# 계획서 — 정기·매칭 모임 동시 운영 + 저장소 분리

작성일: 2026-06-11
상태: **검토 대기 (승인 전, 코드 미변경)**

## 0. 목표 (사용자 요구)
- 정기모임(자체전)은 지금처럼 **팀편성 + 매치테이블** 유지.
- **매칭모임과 정기모임을 같은 시기에 동시에**, **같은 날짜라도** 각각 따로 운영.
- 두 모임의 **참여인원·팀·매치 데이터가 절대 섞이지 않게**, 저장소를 종류별로 분리.

---

## 1. 지금 구조 (왜 큰 작업인가)
이 앱은 **모든 모임 데이터를 "날짜(date)"를 열쇠로** 저장하고, **"현재 모임 1개"** 에 모든 기능을 묶어 동작한다.

### 컬렉션과 키 (전부 날짜 기준)
| 데이터 | 위치 | 문서 키 / 핵심 필드 |
|---|---|---|
| 모임 원본 | `meetings/{date}` | 문서 ID = **날짜** |
| 현재 모임(미러) | `settings/meeting_schedule_v2` | **단 1개** (가장 가까운 모임) |
| 신청 | `registrations/{date}_{memberId}` | 필드 `meetingDate` = 날짜 |
| 선정/출석 명단 | `weekly_session/{date}_{memberId}` | 필드 `date` = 날짜 |
| 팀 편성 | `team_drafts/{자동}` | 필드 `meetingDate` = 날짜 |
| 매치표 | `match_schedules/{자동}` | 필드 `meetingDate` = 날짜, **최신 1개만 표시** |

### 서버(Cloud Functions, `functions/index.js`)
- 자동 노쇼/대기 승급: `meetings/{date}`, 미러, `weekly_session/{date}_{mid}`, `registrations where meetingDate==date` 사용.
- 정기 자동 생성(`generateRecurringMeeting`): `meetings/{date}` + 미러에 기록.

### 관리자 전용 페이지(별도 HTML, 같은 컬렉션 공유)
- `attendance.html`(출석 관리), `team-maker.html`(팀), `match.html`(매치). 공유 부품 `tab-meetings.js`는 member·attendance 양쪽 사용.

### 문제의 핵심
- **같은 날짜 = 같은 문서 키** → 같은 날 모임 2개를 만들 수 없음(`meetings/{date}` 충돌).
- 신청·선정·팀·매치가 전부 날짜로 묶여 → **같은 날이면 서로 덮어씀**.
- "현재 모임"이 1개 → 정기/매칭 **동시 관리 불가**.

### 이미 있는 발판 (재사용 가능)
- `tab-attend.js`에 **`selectedMeeting`(선택한 모임)** 개념이 이미 있어, 활성 모임이 아닌 모임도 골라 명단을 본다 → "관리 대상 모임 선택"으로 확장 가능.
- 홈은 이미 **정기/매칭 두 카드**로 분리 표시 중.

## 2. 설계 방향
### 핵심: 모임 식별자 `meetingId` 도입
- **정기(self): `meetingId = 날짜`** → 기존 데이터 키 그대로 = **이전(마이그레이션) 불필요, 위험 최소**.
- **매칭(match): `meetingId = 날짜__match`** → 별도 키 = 같은 날도 충돌 없음, "저장소 따로".

### 데이터 키 규칙 (날짜 → meetingId)
- `meetings/{meetingId}` (정기는 그대로 날짜)
- `registrations/{meetingId}_{memberId}` + 필드 `meetingId`
- `weekly_session/{meetingId}_{memberId}` + 필드 `meetingId`
  - ⚠️ 기존 `date` 필드는 **진짜 달력 날짜로 유지**(월 계산·날짜 표시에 쓰임). 식별은 새 `meetingId` 필드로.
- `team_drafts` + 필드 `meetingId`
- `match_schedules` + 필드 `meetingId`
- 미러 2개: `meeting_schedule_v2`(정기) + **신규 `meeting_schedule_match`(매칭)**.

### 현재 모임 2개
- 종류별 "가장 가까운 예정 모임" 2개 산출 → `activeSelf`, `activeMatch`.
- 관리 화면은 **선택한 모임(selectedMeeting)** 기준으로 동작(홈 카드 탭 → 그 모임 선택).

---

## 3. 단계별 실행 계획 (안전 순서)
각 단계는 **독립 검증** 후 다음으로. 단계마다 커밋 → 문제 시 직전 커밋 복구.

### Phase 0 — 백업 & 사전 가드
- `admin-data-tool.html`로 **전체 백업**(필수).
- 중복 검사: "같은 날 + 같은 종류"만 막고, "같은 날 + 다른 종류"는 허용하도록 준비.
- 변경: 없음(준비/백업).  위험: 없음.

### Phase 1 — 모임 식별자(meetingId) 기반
- 공통 헬퍼 `meetingId(m)` 추가 (`utils.js` 또는 `firebase-init.js`): self→date, match→`${date}__match`.
- `handlers-meetings.js`: 저장 문서 ID = `meetingId`, 중복 검사 = (날짜+종류), 같은 날 다른 종류 허용.
- 기존 정기 데이터: 키 불변 → 영향 없음. 기존 매칭 모임(있다면)만 새 키로 이전.
- 파일: `handlers-meetings.js`, `utils.js`. 위험: **낮음**.

### Phase 2 — 현재 모임 2개 + 미러 2개
- 활성 모임 종류별 산출(`activeSelf`/`activeMatch`).
- `handlers-meetings.js syncMirror`: 종류에 맞는 미러(`_v2`/`_match`)에 기록. 삭제 시 동일.
- `member.html`: 미러 2개 구독, 홈 두 카드에 각각 연결.
- 파일: `handlers-meetings.js`, `member.html`. 위험: **중간**.

### Phase 3 — 데이터 라우팅을 meetingId로 (가장 민감)
- `registrations`·`weekly_session` 문서 ID/필드, `team_drafts.meetingId`, `match_schedules.meetingId`를 meetingId 기준으로.
- `use-attend.js`/`use-team.js`/`use-match.js`가 **선택 모임의 meetingId**로 질의.
- `tab-attend.js`의 기존 `selectedMeeting` 확장 → 관리 대상 모임 명시 선택.
- `firestore.indexes.json`: `meetingId + status + registeredAt` 복합 색인 추가(대기 승급용).
- 파일: `use-attend.js`, `use-team.js`, `use-match.js`, `handlers-attend*.js`, `handlers-registration.js`, `handlers-team.js`, `team-storage.js`, `handlers-match.js`, `firestore.indexes.json`. 위험: **높음** → 컬렉션별로 쪼개 검증.

### Phase 4 — 화면 라우팅(홈/모임탭)
- 홈 두 카드 탭 → 해당 종류 모임을 선택 모임으로 설정 후 모임탭 진입.
- 모임탭: 선택 모임이 매칭이면 `출석·명단`, 정기면 `출석·팀·매치`(현재 종류 분기 재사용).
- 파일: `member.html`, `tab-home.js`, `tab-attend.js`. 위험: **중간**.

### Phase 5 — 서버(Cloud Functions)
- 자동 노쇼·대기 승급: 미러 2개 각각 처리, `weekly_session`/`registrations` 키 = meetingId.
- 정기 자동 생성: 정기 미러/키로.
- **`firebase deploy --only functions` 별도 배포** + 색인 배포.
- 파일: `functions/index.js`, `firestore.indexes.json`. 위험: **중간**(서버, 별도 배포).

### Phase 6 — 관리자 전용 페이지 정렬
- `attendance.html`·`team-maker.html`·`match.html`에 같은 meetingId 규칙 적용.
- 공유 부품(`tab-meetings.js`)은 양쪽 동시 반영 — 안전 확인.
- 파일: 위 3개 HTML + 공유 .js. 위험: **중간**.

---

## 4. 데이터 마이그레이션
- **정기(self): 불필요** (키 = 날짜 그대로).
- **기존 매칭 모임/데이터**가 있으면 새 키(`날짜__match`)로 이전 — `migrate.html` 또는 `admin-data-tool.html`에 일회성 스크립트.
- 마이그레이션 전 **백업 필수**.

## 5. 위험과 안전장치
- 출석·신청·팀은 **운영 중 민감 데이터**. Phase별 커밋·검증, 중간 단계는 과도기 표시 허용.
- 정기 데이터 키 불변 → 가장 흔한 경우의 위험 최소화.
- 롤백: Phase 커밋 단위 복구 + 백업 복원.

## 6. 검증 체크리스트(공통)
- 같은 날 정기+매칭 동시 생성 가능.
- 각 모임의 신청·선정·출석이 서로 안 섞임.
- 정기: 팀편성·매치표 정상 / 매칭: 출석·명단만.
- 홈 두 카드가 각자 모임으로 진입.
- 서버 자동 노쇼·대기 승급이 모임별로 정확.
- PC·안드로이드·iOS 좌우 스크롤·잘림 없음.

---

## 7. 권장 진행
Phase 0(백업) → Phase 1 → 검증 → 승인 → Phase 2 … 순.
각 단계 시작 전 그 단계의 세부 변경을 다시 보고하고 승인받는다.
